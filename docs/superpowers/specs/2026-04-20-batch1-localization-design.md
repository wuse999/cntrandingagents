# cntrandingagents 首批汉化里程碑设计

## 1. 背景

`cntrandingagents` 当前已经完成仓库治理基线、协作规则和自动执行文档清单，但正式汉化主线还没有落下第一个面向用户的可交付批次。为了确保后续能持续跟进官方上游，本轮只做“用户可见文案层”的首批汉化，不碰交易逻辑、数据流和图编排行为。

用户已明确要求本轮无需中途干预，因此本设计直接采用推荐方案并视为预批准执行。

## 2. 目标

- 让仓库首页 `README.md` 具备完整中文阅读体验。
- 让 CLI 首次上手、交互提问、状态展示、收尾提示切换为中文口径。
- 把本轮动作完整写入执行记录，保持后续可追踪、可复用。
- 保持行为不变，只修改用户可见文本层。

## 3. 范围与非目标

### 3.1 范围

- `README.md`
- `cli/main.py`
- `cli/utils.py`
- `cli/announcements.py`
- `cli/config.py`
- 本轮对应的执行记录、设计和计划文档

### 3.2 非目标

- 不修改 `tradingagents` 图执行逻辑
- 不修改数据源选择、模型选择和策略行为
- 不引入 A 股支持或任何功能增强
- 不重构 CLI 架构，只改展示文案

## 4. 备选方案

### 方案 A：只汉化 README

- 优点：改动最小，几乎没有运行风险。
- 缺点：CLI 仍是英文，上手体验割裂，无法形成“已经可用的中文版”感受。

### 方案 B：README + CLI 用户可见文案一并汉化

- 优点：首批交付完整，用户第一次 clone、安装、启动 CLI 都能看到中文。
- 缺点：改动面比 README-only 稍大，需要补一次基础运行校验。

### 方案 C：全仓所有字符串一次性汉化

- 优点：表面进度最快。
- 缺点：范围失控，极易误碰内部 prompt、日志、测试与行为相关文本，不适合作为第一批。

## 5. 采用方案

采用方案 B。

原因：

- 它在“首批可交付体验”和“同步官方可控性”之间最平衡。
- 仍然严格停留在用户可见文案层。
- 做完后可以自然进入第二批，例如补 package 使用说明、示例脚本和更多 CLI 细节。

## 6. 设计要点

### 6.1 README 设计

- 保留徽章、链接、图片、命令示例和项目名 `TradingAgents`。
- 将说明文字、章节标题、安装说明、CLI 使用说明、贡献说明和引用说明改为中文。
- 不更改代码块内容，避免把命令示例翻坏。

### 6.2 CLI 设计

- 将欢迎语、步骤标题、输入提示、错误提示、保存报告提示、完成提示、报告区块标题翻译为中文。
- 保留内部值和程序逻辑，例如 `AnalystType`、provider key、模型 ID、输出语言值等，不改程序使用的数据结构。
- 对需要展示给用户的选项，优先采用“中文主文案 + 必要英文专有名词”的方式。

### 6.3 执行记录设计

- 本轮新增一份当天执行记录，记录 README 与 CLI 首批汉化动作。
- 如果验证中出现新的乱码、行尾或 CLI 兼容性问题，回收到长期踩坑台账。

## 7. 风险与处理

- 风险 1：CLI 选项值被误翻译，导致程序逻辑出错。
  处理：只翻译 display 文本，不改程序依赖的 value。

- 风险 2：README 中文化过程中误改命令、文件名或链接。
  处理：代码块、路径、URL 和项目名保持原样。

- 风险 3：Windows 本地镜像显示正常，但 Ubuntu 正式主工作树存在编码或行尾差异。
  处理：正式收尾前在 Ubuntu 做 Git 与基础运行复核。

## 8. 验证策略

- 运行 UTF-8 spot-check，确认新增与修改文档没有乱码。
- 运行 `python -m compileall cli main.py` 或等价校验，确保 CLI 文案修改没有引入语法错误。
- 运行 `python -m cli.main --help` 或等价命令查看帮助输出是否可用。
- 在 Ubuntu 正式主工作树检查 `git status -sb`、`git diff --name-only` 并完成正式提交与推送。

## 9. 一句话结论

本轮首批汉化只做 `README` 与 CLI 用户可见文案，把 `cntrandingagents` 从“治理已就绪”推进到“已经有第一批可直接使用的中文版入口”。
