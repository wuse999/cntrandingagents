# cntrandingagents 第二批代码注释与提示词汉化设计

## 1. 背景

`cntrandingagents` 已完成首批 README 与 CLI 用户可见文案的汉化，也已经建立文档治理、执行记录和双端仓库协作基线。本轮任务继续向代码内部推进，但仍严格保持“仅汉化、不加功能”的边界，聚焦尚未完成的英文注释、文档字符串与提示词。

用户已明确要求本轮工作无需中途干预，且当前项目仅做汉化，不叠加 A 股适配或其他功能增强。因此本设计直接按推荐方案落地并视为预先批准执行。

## 2. 目标

- 将 `tradingagents` 与 `cli` 中剩余英文代码注释、文档字符串和提示词改写为中文。
- 保持程序逻辑、状态键、模型 ID、工具名、配置键、供应商标识和调用参数不变。
- 对直接影响 LLM 输出的系统提示词、辩论提示词、提取提示词完成中文化，同时保留必要的结构化指令。
- 将本轮过程、验证结果和新增踩坑回收进既有文档体系，方便后续持续同步上游。

## 3. 范围与非目标

### 3.1 范围

- `tradingagents/agents/` 下分析师、研究员、管理者、风控、交易员相关提示词。
- `tradingagents/agents/utils/` 下工具说明、状态说明、记忆模块注释。
- `tradingagents/dataflows/` 下数据源封装的注释、文档字符串、说明性返回文本。
- `tradingagents/graph/` 下图编排、反思、信号提取相关注释与提示词。
- `tradingagents/llm_clients/` 下客户端封装注释与文档字符串。
- `cli/` 下尚未汉化的代码注释与技术说明。
- 本轮对应的设计文档、计划文档、执行记录与长期踩坑回收。

### 3.2 非目标

- 不修改任何业务逻辑、数据流转逻辑或图执行顺序。
- 不引入中国 A 股支持、Tushare 接口或新供应商。
- 不调整配置键、枚举值、工具函数名、供应商名、模型名或内部路由字符串。
- 不为了汉化而重构模块结构。

## 4. 备选方案

### 方案 A：只翻注释，不翻提示词

- 优点：对 LLM 行为影响最小。
- 缺点：模型仍会持续产出英文报告或英文中间内容，与当前中文化目标不一致。

### 方案 B：注释、文档字符串、提示词分层汉化

- 优点：既能完成代码层中文化，又能控制风险边界，明确哪些字符串必须保留原值。
- 缺点：需要逐文件判断提示词与内部常量的边界，执行量较大。

### 方案 C：对所有英文字符串做全量替换

- 优点：表面推进最快。
- 缺点：极易误改内部键名、模型枚举、供应商标识、错误消息结构和工具调用参数，后续很难同步官方。

## 5. 采用方案

采用方案 B。

原因：

- 它最符合“长期稳、可控、可持续跟随官方”的仓库定位。
- 能把用户真正关心的中文体验推进到提示词层，同时避免误动内部协议字符串。
- 便于未来继续把官方更新按批次同步并复核。

## 6. 设计要点

### 6.1 注释与文档字符串

- 只翻说明性内容，不改函数名、类名、参数名、返回结构。
- `Args`、`Returns`、示例说明统一改为中文口径，但保留代码字面值和格式约束，例如 `YYYY-MM-DD`、`BUY`、`SELL`。
- 对使用 `Annotated[str, "..."]` 的工具参数描述同步中文化，因为它们属于模型可见说明。

### 6.2 提示词

- 系统提示词、研究提示词、辩论提示词、裁决提示词、信号提取提示词统一改为中文。
- 保留必要的结构化英文枚举或硬性标记，例如 `FINAL TRANSACTION PROPOSAL: **BUY/HOLD/SELL**`、评级枚举 `BUY/OVERWEIGHT/HOLD/UNDERWEIGHT/SELL`，以避免破坏后续解析逻辑。
- 对“工具名必须精确使用”的说明，只翻自然语言，不翻工具函数名或指标参数名。

### 6.3 返回文本与报头

- 数据层返回的解释性报头和错误提示可汉化。
- 真正被程序解析的字段名、CSV 列名、JSON 键名、HTTP 参数名保持不变。

## 7. 风险与处理

- 风险 1：误翻内部关键字符串，导致路由、状态判断或结果提取失败。
  处理：只翻自然语言层；凡是被代码条件判断、外部 API、模型/供应商识别依赖的字面值一律不改。

- 风险 2：提示词汉化后改变模型输出习惯，影响后续判断。
  处理：保留关键终止标记、评级枚举和必要英文关键词；验证 `SignalProcessor` 与主流程的提取逻辑不受影响。

- 风险 3：Windows 本地查看正常，但 Ubuntu 正式主工作树出现编码或换行差异。
  处理：继续采用 Windows 编辑、Ubuntu 正式验证与提交的流程，并用 UTF-8 与编译校验双重确认。

## 8. 验证策略

- 使用 `python -m compileall cli tradingagents main.py` 做语法校验。
- 对关键文件执行 `py_compile` 或定向导入，确认字符串调整未引入语法错误。
- 使用 `rg` 抽检高风险提示词和关键硬编码标记仍然存在。
- 在 Ubuntu 正式主工作树复核 Git 状态、编译结果并完成正式提交推送。

## 9. 一句话结论

本轮将 `cntrandingagents` 从“首层中文化”继续推进到“代码说明层与智能体提示词层中文化”，但仍坚持只动文案、不动逻辑，为后续长期同步官方保留稳定底座。
