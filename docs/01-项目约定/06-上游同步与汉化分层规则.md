# 上游同步与汉化分层规则

## 1. 文档用途

- 固定 `TradingAgents` 官方上游与 `cntrandingagents` 汉化主线之间的分层规则。
- 防止汉化仓与官方仓纠缠成一团，导致后续难以同步。

## 2. 为什么要分层

- 官方项目会持续更新。
- 汉化仓必须长期可跟随官方同步。
- 若直接在官方镜像上累计大量汉化改动，后续同步冲突会越来越大。
- 分层后，`raw` 负责官方镜像，`cntrandingagents` 负责汉化主线，职责清晰。

## 3. `raw` 的维护规则

- `raw` 只负责同步官方 `TradingAgents`。
- `raw` 不承载汉化主线。
- `raw` 中不应长期保留你自己的汉化改动。
- `raw` 可以用于读取、对照、Diff 和准备同步。

## 4. `cntrandingagents` 的维护规则

- `cntrandingagents` 只负责汉化主线。
- 任何中文化改动都进入 `cntrandingagents`。
- `cntrandingagents` 默认不承担功能增强。
- `cntrandingagents` 必须保持“差异尽量集中在文案层”的结构。

## 5. 官方更新进入本项目的路径

1. 先同步 `raw` 到最新官方状态。
2. 比对 `raw` 与 `cntrandingagents` 的差异。
3. 判断官方更新是否影响文档、CLI 文案、帮助文本或对外展示文案。
4. 把必要同步内容引入 `cntrandingagents`。
5. 补齐对应中文化。
6. 完成验证后再提交与发布。

## 6. 同步时先看什么

- README 变化
- CLI 帮助和提示文案变化
- 安装方式变化
- 模型支持说明变化
- 项目结构变化
- 用户可见输出变化

## 7. 同步后允许保留的差异

- 中文 README
- 中文帮助文本
- 中文注释和中文说明
- 汉化版专用术语表与治理文档
- 不改变行为的中文展示文案

## 8. 同步后不应保留的差异

- 功能性重构
- 数据流逻辑差异
- Prompt 行为差异
- 图编排差异
- 与上游无关的“顺手优化”

## 9. 同步冲突处理原则

- 优先保持官方行为不变。
- 优先保留官方新逻辑。
- 中文化只覆盖用户看得到的文案层。
- 如同步会引发行为分叉，记录下来，不在本仓继续扩大修改。

## 10. 什么时候可以跳过同步

- 官方更新完全与文档、CLI、帮助文本无关。
- 当前阶段只在整理治理文档，还未开始正式汉化主线。
- 官方更新会引入大规模功能变更，而本轮不准备吸收。

## 11. 记录要求

- 每轮官方同步必须在 `docs/03-上游同步` 留下记录。
- 说明上游版本、同步范围、冲突文件、处理方式和验证结果。

## 12. 一句话结论

`raw` 是官方镜像，`cntrandingagents` 是汉化主线；先同步官方，再补汉化，永远不要把两者混成一条线。
