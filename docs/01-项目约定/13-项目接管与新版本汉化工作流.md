# 项目接管与新版本汉化工作流

## 1. 文档用途

- 作为 `cntrandingagents` 的单入口接管说明，方便下一个会话或协作者不翻历史聊天也能直接接手。
- 统一说明“当前项目做到哪一步、下一轮应怎么接、上游新版本该怎么汉化并推回仓库”。
- 把会话交接需要的背景、边界、命令和收尾动作固定成长期规则。

## 2. 项目一句话结论

`cntrandingagents` 是官方 `TauricResearch/TradingAgents` 的中文汉化专用仓，当前唯一目标是持续跟进官方版本并补齐中文化，不做 A 股扩展、不做功能增强、不制造与上游分叉的行为差异。

## 3. 当前项目基线

### 3.1 已固定的协作模式

- Windows：本地镜像、应急副本、文档入口，不是默认主工作树。
- Ubuntu：唯一正式主工作树。
- GitHub：主仓与主发布仓。
- Gitee：镜像仓与国内发布源。
- `raw`：官方上游镜像，只做读取、Diff、对照和同步准备。
- `cntrandingagents`：正式汉化主线仓。

### 3.2 已完成的长期治理基础

- `docs/01-项目约定` 已建立完整治理文档体系。
- `docs/02-执行记录` 已建立执行留痕、模板与长期踩坑台账。
- `docs/03-上游同步` 已建立同步记录入口与模板。
- `docs/01-项目约定/11-AI自动执行文档与检查清单.md` 已固化默认执行流程。
- `tools/cleanup_windows_mirror_backups.ps1` 已建立 Windows 本地镜像备份保留脚本，默认保留最近 `3` 份完整快照。

### 3.3 已完成的汉化成果

- 首批：README 与 CLI 用户可见文案已汉化。
- 第二批：代码注释、文档字符串、提示词与部分数据层自然语言说明已汉化。
- PNG 资源已完成一轮中文化落地，当前已处理：
  - `assets/analyst.png`
  - `assets/researcher.png`
  - `assets/risk.png`
  - `assets/schema.png`
  - `assets/trader.png`
  - `assets/wechat.png`
- 与官方行为强绑定的硬标记已明确保留英文，不随汉化翻译。

### 3.4 当前必须继续保持不变的部分

- 不改核心行为、不改图编排、不改数据流逻辑。
- 不在本仓内叠加 A 股支持。
- 不把长期敏感信息写入仓库。
- 不把 Windows 镜像当正式 Git 收口面。

## 6. 新版本汉化更新的标准流程

### 6.1 先判断任务类型

先确认本轮到底属于哪一类：

- 官方上游有新版本，需要同步后补汉化。
- 官方代码不变，只是在补扫漏网英文。
- 只是在更新治理文档或执行记录。

若任务已经超出“汉化或治理文档”范围，应停止在本仓继续扩写。

### 6.2 先更新上游对照面

优先更新 `raw`，确保对照基线是新的官方版本。

推荐动作：

```powershell
cd D:\tranding\tranding_agents_cn\raw
git status -sb
git remote -v
git fetch origin --prune
git pull --ff-only origin main
```

如 `raw` 的默认远端命名不同，以实际配置为准，但其角色仍然是“官方镜像，不承载汉化主线”。

### 6.3 在汉化仓判断哪些变化需要吸收

重点检查：

- README、安装说明、使用说明是否有变化
- CLI 帮助与用户可见提示是否有变化
- Prompt、说明文案、报告标题、对外文字是否有变化
- PNG 或流程图中是否新增英文文本
- 项目结构是否有变化，导致原有汉化点位置移动

不需要重新汉化整仓，应只吸收与本轮官方更新直接相关的部分。

### 6.4 开工前先补记录

每轮正式工作开始前，至少补两类文档之一：

- `docs/02-执行记录/YYYY-MM-DD-事项名-执行记录.md`
- 如涉及上游更新，再补 `docs/03-上游同步/YYYY-MM-DD-事项名.md`

记录中至少写明：

- 本轮目标
- 对照基线
- 改动范围
- 验证策略
- 当前阻塞和后续待办

### 6.5 实施汉化时的基本原则

- 只改用户可见文案层。
- 内部逻辑、函数行为、图编排和数据流逻辑不随汉化改变。
- Prompt 中可翻译自然语言，但被解析逻辑依赖的英文硬标记必须原样保留。
- CLI 参数名、环境变量名、模型 ID、工具名、内部状态键与文件路径不随汉化翻译。
- PNG 汉化优先保持原结构、原层级、原观感，不用大白块遮罩长期凑合。

### 6.6 默认验证顺序

1. Windows 本地 UTF-8 读取与 Diff 复核。
2. 轻量语法验证：

```powershell
python -m compileall cli tradingagents main.py
```

3. 关键硬标记抽检：

```powershell
rg -n "FINAL TRANSACTION PROPOSAL|BUY/OVERWEIGHT/HOLD/UNDERWEIGHT/SELL" tradingagents
```

4. 如涉及 PNG，做肉眼复核，优先检查结构、字重、对齐和背景是否被破坏。
5. 回 Ubuntu 正式主工作树做正式 Git 收口与最终验证。

### 6.7 Ubuntu 正式收口步骤

推荐动作：

```bash
cd ~/projects/cntrandingagents
export GIT_SSH_COMMAND='/mnt/c/Windows/System32/OpenSSH/ssh.exe -o BatchMode=yes -o ConnectTimeout=10'
git status -sb
git fetch origin --prune
git fetch upstream --prune
git fetch gitee --prune
```

随后：

1. 仅同步本轮真实改动文件到 Ubuntu。
2. 在 Ubuntu 复核 `git status -sb`、关键文件内容与必要验证命令。
3. 提交到 `main`。
4. 推送到 `origin main`。
5. 推送到 `gitee main`。
6. 再核对 Ubuntu `HEAD`、`origin/main`、`gitee/main` 是否一致。

### 6.8 Windows 镜像收尾

正式推送完成后：

1. 把 Windows 本地镜像快进到最新 `origin/main`。
2. 确认 Windows 镜像工作树回到干净状态。
3. 若本轮产生了镜像对齐备份，执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\cleanup_windows_mirror_backups.ps1 -WhatIf
```

必要时再正式执行清理。

## 7. 哪些内容要汉化，哪些不要动

### 7.1 优先汉化的内容

- README 与文档中的用户说明
- CLI 用户可见提示、帮助文案、错误提示
- 代码注释与文档字符串
- Prompt 中不影响解析的自然语言
- 报告标题、说明性字段、输出报头
- PNG 等可见资源中的英文文字

### 7.2 禁止随意翻译的内容

- `FINAL TRANSACTION PROPOSAL`
- `BUY/OVERWEIGHT/HOLD/UNDERWEIGHT/SELL`
- 内部状态键、工具名、函数名、模块名
- 环境变量名、模型名、Provider 名
- CLI 参数名、配置键、文件路径
- 与上游逻辑强绑定的结构化英文字面值

## 8. 高风险点清单

- 最大风险不是“漏翻一句英文”，而是误翻了解析依赖硬标记。
- 最大流程风险不是“少写一份文档”，而是在 Windows 镜像上直接做正式 Git 收口。
- 最大编码风险不是“终端看起来像乱码”，而是未经 UTF-8 复核就直接覆盖中文文件。
- 最大同步风险不是“少拉一次上游”，而是把汉化和功能增强混在一起，导致以后无法跟上官方。
- 最大安全风险不是“文档不够详细”，而是把 SSH 私钥、PAT 或其他敏感凭据明文写进仓库。
