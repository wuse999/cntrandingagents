# AI 自动执行文档与检查清单

## 1. 文档用途

- 为进入 `cntrandingagents` 的 AI、自动化脚本和协作者提供统一执行顺序。
- 把“自动记录文档、自动记录踩坑、自动迭代项目经验、自动检查、自动规避中文乱码”固定成默认动作。
- 让每一轮汉化、治理文档更新和上游同步都能留痕、可复盘、可延续。

## 2. 适用范围

- 适用于 `cntrandingagents` 的汉化、治理文档维护、上游同步前后检查。
- 不适用于功能增强、行为改造、A 股数据层扩展等新项目需求。
- 如任务已超出汉化范围，应先回到 [01-汉化范围与非目标.md](d:/tranding/tranding_agents_cn/cntrandingagents/docs/01-项目约定/01-汉化范围与非目标.md) 重新确认边界。

## 3. 默认执行原则

- 先读规则，再动文件。
- 先判断当前机器角色，再执行 Git。
- 先确认 UTF-8，再复制、改写或覆盖中文内容。
- 先记录事实，再写结论。
- 重复出现的问题必须进入长期踩坑台账。
- 可以稳定复用的经验必须回写治理文档或模板，而不是只留在聊天记录里。

## 5. 开始执行前的固定检查

### 5.1 Windows 本地镜像检查

在 Windows 本地查看或整理文档前，先执行：

```powershell
$OutputEncoding = [Console]::OutputEncoding = [System.Text.UTF8Encoding]::new($false)
[Console]::InputEncoding = [System.Text.UTF8Encoding]::new($false)
Get-Location
hostname
git status -sb
git branch -vv
git remote -v
```

说明：

- Windows 目录默认是本地镜像、应急副本和文档入口。
- Windows 侧可以做查看、比对、草拟和预编辑。
- 正式 `git commit`、`git push`、发布和主线收口动作，必须回到 Ubuntu 正式主工作树执行。

### 5.2 Ubuntu 正式主工作树检查

进入 Ubuntu 正式主工作树前，先执行：

```bash
cd ~/projects/cntrandingagents
pwd
hostname
git status -sb
git branch -vv
git remote -v
git fetch origin --prune
git fetch upstream --prune
git fetch gitee --prune
locale
```

说明：

- Ubuntu 是唯一正式主工作树。
- 所有正式 Git 动作、正式验证、正式推送与发布，都以 Ubuntu 结果为准。

## 6. 默认自动执行流程

### 6.1 锁定任务边界

开始前必须确认：

- 本轮是否仍然只做汉化或治理文档维护。
- 是否会碰到 [07-允许修改的文件类型与改动边界.md](d:/tranding/tranding_agents_cn/cntrandingagents/docs/01-项目约定/07-允许修改的文件类型与改动边界.md) 中禁止修改的区域。
- 是否涉及官方同步；如涉及，后续还必须更新 `docs/03-上游同步`。

### 6.2 建立本轮执行记录

- 每一轮实际工作都必须先在 `docs/02-执行记录` 建立或补全一份记录。
- 推荐命名：`YYYY-MM-DD-事项名-执行记录.md`
- 如当日记录不存在，先按 [模板-汉化执行记录.md](d:/tranding/tranding_agents_cn/cntrandingagents/docs/02-执行记录/模板-汉化执行记录.md) 创建。
- 先填“背景、目标、环境、编码检查”四块，再开始改文件。

### 6.3 扫描受影响文件

扫描前先判断这轮改动落在哪一层：

- 官方对照层：`raw`
- 汉化主线层：`cntrandingagents`
- 治理文档层：`docs/01-项目约定`
- 执行记录层：`docs/02-执行记录`
- 上游同步层：`docs/03-上游同步`

推荐命令：

```powershell
git diff --name-only
rg --files docs
rg -n "关键词" docs raw
```

### 6.4 执行修改

- 优先保持官方行为不变，只改用户可见文案层。
- 新增文档时，同时补齐对应索引或 README。
- 修改术语时，先回看 [08-翻译术语表与风格约定.md](d:/tranding/tranding_agents_cn/cntrandingagents/docs/01-项目约定/08-翻译术语表与风格约定.md)。
- 如需在 Windows 预编辑，收尾前仍要在 Ubuntu 正式主工作树复核一遍。

### 6.5 自动记录踩坑

出现下面任一情况时，必须追加到 [00-长期踩坑与经验回收.md](d:/tranding/tranding_agents_cn/cntrandingagents/docs/02-执行记录/00-长期踩坑与经验回收.md)：

- 同类问题重复出现两次及以上。
- 问题涉及 Git、远端、权限、编码、发布、同步策略。
- 问题虽然已临时解决，但下次仍大概率再次踩到。

记录时必须写明：

- 触发场景
- 现象
- 临时处理
- 后续应回写到哪份长期文档
- 当前状态是“观察中”还是“已固化”

### 6.6 自动沉淀项目经验

当某条经验已经稳定可复用时，不应只停留在执行记录中，应升级到：

- `docs/01-项目约定` 的长期规则
- `docs/02-执行记录/模板-汉化执行记录.md` 的固定字段
- `docs/03-上游同步` 的同步模板或流程说明

判断标准：

- 已重复使用至少两次
- 会影响多人协作或跨机器一致性
- 会影响后续上游同步、发布或中文编码稳定性

### 6.7 自动检查

正式收尾前至少执行以下检查：

```powershell
$OutputEncoding = [Console]::OutputEncoding = [System.Text.UTF8Encoding]::new($false)
git status -sb
git diff --name-only
rg -n "gh[p]_[A-Za-z0-9_]+|github[_]pat_[A-Za-z0-9_]+|-----BEGIN[ ]+.*PRIVATE KEY-----|xox[p]-[A-Za-z0-9-]+" docs .
Get-Content -Encoding UTF8 -Path .\docs\00-总索引.md
```

如在 Ubuntu 正式检查，则执行：

```bash
git status -sb
git diff --name-only
rg -n "gh[p]_[A-Za-z0-9_]+|github[_]pat_[A-Za-z0-9_]+|-----BEGIN[ ]+.*PRIVATE KEY-----|xox[p]-[A-Za-z0-9-]+" docs .
sed -n '1,80p' docs/00-总索引.md
```

检查目标：

- 改动范围仍然符合“只做汉化 / 治理文档”边界。
- 中文文件可正常显示，不存在新的乱码。
- 没有把私人令牌、私钥等敏感明文写进仓库。
- 新增文档已经被索引或 README 收录。

若正式运行验证受环境依赖或网络拉包超时影响，则按以下降级顺序继续：

1. 先执行 `python -m py_compile` 或 `python -m compileall`，确保语法层通过。
2. 再执行轻量模块导入验证，确认本轮直接修改的模块仍可被解释器加载。
3. 再优先运行低依赖、与本轮改动直接相关的已有测试。
4. 完整 CLI 启动、全量测试和重依赖验证只在依赖已备齐的正式环境中执行。

### 6.8 正式收尾

收尾时必须完成：

1. 补完当日执行记录。
2. 如出现新坑，补 `00-长期踩坑与经验回收.md`。
3. 如形成新规则，补 `docs/01-项目约定`。
4. 如涉及官方同步，补 `docs/03-上游同步`。
5. 正式 Git 动作回到 Ubuntu 执行。

## 7. 中文编码与乱码规避规则

- 仓内 Markdown、代码与配置文件默认按 UTF-8 处理。
- Windows 下查看中文文件时，必须显式使用 `Get-Content -Encoding UTF8`。
- 避免使用 PowerShell 默认重定向或默认 `Out-File` 去覆盖中文文件，以免写成非 UTF-8 编码。
- 若终端显示疑似乱码，不要直接依据乱码输出回写文件，应以 IDE 文件内容或 Ubuntu UTF-8 环境为准。
- 若 Windows 本地镜像出现 `LF will be replaced by CRLF` 之类的行尾提示，不要直接在本地定案，先回 Ubuntu 正式主工作树复核。
- 如无法确认某份中文文件当前编码，先停止覆盖，先做只读检查，不要边猜边改。

## 8. 哪些经验要升级成长期规则

出现以下任一情形，必须从执行记录升级到治理文档：

- 同类坑再次出现。
- 某条经验会影响所有后续汉化轮次。
- 该经验涉及 Windows 与 Ubuntu 协作口径。
- 该经验涉及上游同步、远端推送、发布或凭据处理。
- 该经验涉及中文编码、终端显示或文件读写稳定性。
